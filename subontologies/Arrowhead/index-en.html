AH Mandatory Core Service IDDc back to ToC or Class ToC

IRI: http://www.w3id.org/ecsel-dr-AH#AH_Mandatory_Core_Service_IDD

Class of all Interface Design Descriptions (IDDs) of mandatory Arrowwhead core services. While the class AH_mandatory_core_service contains the mandatory core services only as instances, this class contains classes of the IDDs of each single mandatory core service as subclasses (not only as instances). Since each AH Local Cloud runs one instance of each mandatory core service, each of these services has to be documented by an AH service documentation, i.e. at least by a SD and a IDD. So the IDD is an individual document for each mandatory core service in each single Local Cloud. Hence, all IDDs of a certain mandatory core service will be individuals of the corresponding IDD class.

AH Service Documentation Documentc back to ToC or Class ToC

IRI: http://www.w3id.org/ecsel-dr-AH#AH_Service_Documentation_Document

Document belonging to the documentation of an Arrowhead Service. There are four types of documents for Arrowhead service documentations: - Service Description - Interface Design Description - Communication Profile - Semantic Profile. The central documents are IDDs. They document the implementation of a certain service based on a certain communication technology (communication, encoding, data format) and a certain information model. The Service Description contains a more general description of the service, without the link to a specific communication technology or information model. Communication Profiles and Semantic Profiles are optional documents for a service documentation. They can be seen as modules of an IDD, since they can describe the communication technolopgy and the information model of a service implementation as external documents. If they exist, the IDD refers to them instead of declaring these contents by themselfes. That is, CP and SP are useful if different service implementations use the same CP and SP - then their IDDs can refer to one and the same CP or SP document.

describes the relation between two documentation documents that one document refers to another document. To our best knowledge, the following relations hold between different types of documentation documents: - an IDD refers to exactly one SD. - an SD is referred to by one or more IDD (each stating a way of implementing the service with a specific technolopgy (Source: Arrowhead Framework Definitions V2.4)) - an IDD refers to at most one CP (if it not refers to a CP, then the IDD contains all CP information by itself) - an IDD refers to n SP, 0 <= n (if it not refers to a SP, then the IDD contains all SP information by itself) - a SySD refers to one or more IDDs (that is, a system produces one or more services). - a SySDD refers to one or more SySD's. refers to CPop back to ToC or Object Property ToC

IRI: http://www.w3id.org/ecsel-dr-AH#refers_to_CP

an Interface Design Description (IDD) can refer to a Communication Profile (CP). No other documentation document can refer to a CP. Hence the domain of this object property is the class of IDDs. An IDD can refer to a CP, but this is not mandatory. CPs are intended to be used in cases in which several services use the same CP. Then the CP information should be extracted to an CP document and the IDDs of all services that use this CP shall refer to this CP. If the CP is unique for a specific service, the CP information can directly be included in the IDD. If the IDD refers to a CP, this must be a unique reference. That is, one IDD must not refer to different CPs. Hence this object property is functional. (If the service could be implemented using different communication technologies, each implementation would need an own IDD!!).
