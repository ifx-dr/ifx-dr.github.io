The term "Digital Factory" is used in different meanings today and should hence not longer be used. It should be replaced by more specific terms like "Digital Factory Planning system" or "Digital Factory Twin". Originally, Digital Factory referred to software tools for computer-aided planning, design, engineering and simulation of production, manufacturing, or assembly systems. These tools supplemented CAD-based tools for factory layout planning and tools for computer-aided tools for production engineering (work plan preparation, production process planning) and were glued together with them to PLM software suites. As today even CAD-based layout planning and process planning tools are "digital" and refer to the factory, the terms "Digital Factory tool" and "Production Engineering tool" more and more converge. At the same time, automation vendors start to use the concept "Digital Factory" to address online monitoring and control tools for production systems. In this meaning, the term is a synonym for "Digital Factory Twin" or "Digital Twin of a Factory". The important difference between these two meanings is the phase in the life cycle in which the tool is used: Manufacturing Planning / Eingineering is conducted mainly in the first two lifecycle phases in which the product (type, class) is created, designed, and engineered, as well as the corresponding production system. The online monitoring and control is conducted during the third phase, in which the product individuals are produced on the existing production system. Hartwig Baumgaertel, 2018-11-12

A field bus is a bus system that connects field devices such as sensors and actuators in a plant for the purpose of communication with an automation device. If several communication participants send their messages over the same line, then it must be specified who (identifier) says what (measured value, command) and when (initiative). There are standardized protocols for this. The first generation of fieldbus technology was developed in the 1980s to replace the parallel wiring of binary signals and analog signal transmission that had been common until then with digital transmission technology. Today, many different fieldbus systems with different characteristics are established on the market. Since 1999, fieldbuses have been standardized worldwide in the IEC 61158 standard (Digital data communication for measurement and control - Fieldbus for use in industrial control systems) and by the IEC 61784 standard. The second generation of fieldbus technology is based on real-time Ethernet (Industrial Ethernet). A fieldbus defines a complete protocol stack over the 7 layers of the OSI layer model. If necessary, not all layers are occupied. Example for first generation fieldbus is Profibus, Example for second generation fieldbus is ProfiNet.

MQTT[1] (Message Queuing Telemetry Transport) is an ISO standard (ISO/IEC PRF 20922)[2] publish-subscribe-based messaging protocol. It works on top of the TCP/IP protocol. It is designed for connections with remote locations where a "small code footprint" is required or the network bandwidth is limited. The publish-subscribe messaging pattern requires a message broker. Andy Stanford-Clark of IBM and Arlen Nipper of Cirrus Link authored the first version of the protocol in 1999.[3] The specification does not specify the meaning of "small code footprint" or the meaning of "limited network bandwidth". Thus, the protocol's availability for use depends on the context. In 2013, IBM submitted MQTT v3.1 to the OASIS specification body with a charter that ensured only minor changes to the specification could be accepted.[4] MQTT-SN[5] is a variation of the main protocol aimed at embedded devices on non-TCP/IP networks, such as ZigBee. Historically, the "MQ" in "MQTT" came from IBM's MQ Series message queuing product line.[6] However, queuing itself is not required to be supported as a standard feature in all situations.[7] Alternative protocols include the Advanced Message Queuing Protocol (AMQP), Streaming Text Oriented Messaging Protocol (STOMP), the IETF Constrained Application Protocol,[8] XMPP,[9][10] DDS,[11] OPC UA,[12] and Web Application Messaging Protocol (WAMP). SCTPc back to ToC or Class ToC

IRI: http://www.w3id.org/ecsel-dr-AT#SCTP

It was proposed by the Internet Engineering Task Force (IETF) as a new transport protocol and published in October 2000 in the standard document RFC 2960 and extended in September 2007 by RFC 4960. An introduction can also be found in RFC 3286. The responsible body at the IETF is the Signaling Transport Working Group, or SIGTRAN for short. As a transport protocol, SCTP is on the same level of the TCP/IP reference model as TCP and UDP (layer 4 of the OSI model). SCTP implements the concept of an association: A connection is set up in which several message data streams are transported in a sequence-preserving manner (but potentially non-sequence-preserving among themselves). In addition, individual datagrams, for example urgent datagrams, can be sent separately and out of sequence, possibly "overtaking" the in-order data streams. SCTP also knows multistreaming and multihoming (one host with several valid IP addresses). Heartbeats are used to actively test for connection failure. Unlike TCP, SCTP is resistant to SYN flooding, a denial-of-service attack in which half-open connections use up the server's resources. It uses a so-called four-way handshake for this purpose. Here, the server does not store any state information in a connection request (INIT packet), but sends it to the client in the form of a cookie (INIT-ACK packet). The client must include this cookie in its response (COOKIE-ECHO packet) and is thus recognized by the server as authorized to establish a connection, which the server confirms to it (COOKIE-ACK packet). A similar procedure is also possible with TCP, see SYN cookies. SCTP was originally defined as a transport protocol for transmitting signalling messages (SS7) from telephone networks over IP networks. During development, the reliability of the protocol was a particular focus. However, SCTP is also suitable for other applications because it combines the advantages of TCP and UDP. An important application based on SCTP is Reliable Server Pooling (RSerPool). SCTP uses similar algorithms as TCP for flow and congestion control, so it behaves neutrally in a mixed network (SCTP and TCP)[1]. Java also provides its own API for SCTP since version 7.
